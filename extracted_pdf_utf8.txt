3.1 How to run the simulator
To start the simulator, first load the image from the provided tar archive and then run the container
by publishing port 8080:
docker load -i mars-iot-simulator-oci.tar
docker run --rm -p 8080:8080 mars-iot-simulator:multiarch_v1
All endpoints are served from the base URLhttp://localhost:8080. The OpenAPI documenta-
tion is available athttp://localhost:8080/docsandhttp://localhost:8080/openapi.json.
You can also verify the simulator and list available devices using:
curl -s http://localhost:8080/health
curl -s http://localhost:8080/api/discovery | jq
3.2 REST sensors
You can list the available REST sensors by calling the sensors endpoint. The response includes the
sensor identifiers used throughout the system:
curl -s http://localhost:8080/api/sensors | jq

3.4 Actuator APIs
Actuator state can be inspected via the actuators endpoint:
curl -s http://localhost:8080/api/actuators | jq
To change a state, POST a JSON payload to the actuator resource, for example:
curl -X POST http://localhost:8080/api/actuators/cooling_fan \
-H ÆContent-Type: application/jsonÆ \

4 What you must deliver
You must design and implement a distributed automation platform that:

1. Collects data from simulated devices (polling and/or stream depending on team size)

4.1 Persistence
òHistorical sensor data persistence isnotrequired.
òThe latest state per sensor may be cached in memory.
òAutomation rulesmustbe persisted, so that they survive service restarts.
Rule persistence can be implemented using any kind of relational database (e.g., Postgres), noSQL
database (e.g., MongoDB), or embedded database (e.g., SQLite). Complex schema design is not
required; a simple rule table/schema is sufficient.
4.2 Authentication
Authentication isnot required. There are no user accounts. The platform may be treated as
single-tenant. All sensors are globally visible: there is no concept of ownership or association.

5.2 Automation engine
The automation engine must supportsimple ifûthen rules.
IF <sensor_name> <operator> <value> [unit]
THEN set <actuator_name> to ON | OFF
Supported operators:<, <=, =, >, >=.
Example:IF greenhouse_temperature > 28░C THEN set cooling_fan to ON
4
Rules must:
òBe evaluated dynamically on event arrival
òBe persisted
òTrigger an actuator state update when condition is met
5.3 Frontend dashboard

but it is not mandatory: static dashboards are also acceptable.
Optional widget approach

toggle, or custom visualizations by data type.
The frontend must also provide a rule management interface and real-time updates via WebSocket
or SSE.
